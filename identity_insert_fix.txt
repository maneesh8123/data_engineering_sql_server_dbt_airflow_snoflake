Short answer: because `ScannedTime` is not a strictly increasing,
 unique watermark — new rows can have the same or earlier timestamp, so `WHERE ScannedTime > MAX(ScannedTime)` can miss them.

Why this happens (concise):
- Timestamps can be identical for multiple rows (same second/millisecond precision).
- Source may insert rows out of order or with backfilled timestamps.
- Timezone/precision differences or truncation (datetime vs datetime2) can make comparisons unreliable.
- If a new row has ScannedTime <= current MAX(ScannedTime), the predicate excludes it.

Recommended fix (composite predicate):
- Use ScannedTime plus ID as a tie-breaker so you include rows with the same timestamp but higher IDs.

Replace your incremental predicate with this (paste into your model inside the `{% if is_incremental() %}` block):

WHERE
  (ScannedTime > (SELECT MAX(ScannedTime) FROM {{ this }}))
  OR
  (ScannedTime = (SELECT MAX(ScannedTime) FROM {{ this }})
   AND ID > (
     SELECT COALESCE(MAX(ID), 0)
     FROM {{ this }}
     WHERE ScannedTime = (SELECT MAX(ScannedTime) FROM {{ this }})
   )
  )

Other options:
- Keep using `ID` only (works when IDs are monotonic).
- Normalize/upgrade `ScannedTime` precision (use datetime2(7)) and ensure consistent timezones.
- Use a dedupe step instead of strict > filtering (select rows where ID not in target).
